# engspec_code: English-First Specification Layer for Any Codebase

## What This Project Does

Every function in **any** codebase (Python, Rust, Go, C++, JS/TS, Java, etc.) gets a parallel `.engspec` specification written in structured English. The spec is detailed enough that an AI agent can re-implement the function from the spec alone — and does so repeatedly during bootstrap to validate the spec is complete.

`engspec_code` manages: the `.engspec` format, the `input.md` → `input_processed.md` pipeline, skeleton `.engspec` generation, the agent swarm for full spec creation + verification, and call graph / composition tools.

---

## The Full Pipeline

```
┌──────────────────────────────────────────────────────────────────┐
│ STEP 1: input.md                                                 │
│   Human-provided (or auto-generated via create-input).           │
│   Contains: project description, test setup, how to run tests,   │
│   examples, notes.                                               │
├──────────────────────────────────────────────────────────────────┤
│ STEP 2: engspec process-input                                    │
│   Claude deep-analyzes the entire repo using input.md.           │
│   Produces TWO outputs:                                          │
│                                                                  │
│   (a) input_processed.md                                         │
│       - Common Workflows (step-by-step code paths)               │
│       - Common Call Graphs (function→function trees)             │
│       - Test Coverage Analysis (per function: tested/not tested) │
│                                                                  │
│   (b) Skeleton .engspec files (one per source file)              │
│       - Function signature + brief purpose for each function     │
│       - Key context: callers, callees, test coverage status      │
│       - Placeholder sections for pre/postconditions, etc.        │
├──────────────────────────────────────────────────────────────────┤
│ STEP 3: Agent swarm (parallel, independent)                      │
│   One agent per .engspec file. Each agent:                       │
│   1. Reads: skeleton .engspec + source file + input_processed.md │
│   2. Fills in full spec for every function                       │
│   3. Runs regeneration verification loop:                        │
│      - Re-implement from spec alone N times                      │
│      - Compare each regeneration against original                │
│      - If any regen misses essential properties → refine spec    │
│      - Repeat until all N regenerations pass                     │
│   4. Returns validated .engspec file                             │
│                                                                  │
│   Agents are fully independent — no coordination needed.         │
├──────────────────────────────────────────────────────────────────┤
│ STEP 4: (optional) engspec_tester                                │
│   Adversarial debate to find bugs and beef up specs further.     │
└──────────────────────────────────────────────────────────────────┘
```

**No tree-sitter or AST parsing libraries needed.** Claude does the "parsing" during Step 2. The skeleton `.engspec` files ARE the parsed output — function signatures, call relationships, and context are all captured in English.

## input.md: Required Project Context

Before running `code→engspec`, the user provides (or generates) an `input.md`.

### Required sections:

```markdown
# Project: <name>

## Description
What this repository does, its purpose, key abstractions, architecture overview.

## Test Environment Setup
How to install dependencies, set up the development environment,
required system packages, Docker/venv/container instructions.

## Running Tests
How to run the test suite. Specific commands, flags, config files.
Examples of running individual tests, test suites, markers.

## Examples
Example usage of the main APIs/CLIs. Input/output examples.
Links to documentation if available.

## Notes
Any other context: known issues, conventions, important directories,
external dependencies, etc.
```

### create-input tool

```
engspec create-input <repo-path>      # Analyze local repo, generate draft input.md
engspec create-input --url <git-url>  # Clone + analyze
```

Reads: README, CONTRIBUTING, Makefile/justfile, CI configs, build files (pyproject.toml, Cargo.toml, package.json, CMakeLists.txt, go.mod), test directories, docstrings, code structure, git history. Produces a draft `input.md` for human review.

## input_processed.md: AI-Generated Deep Analysis

Generated by `engspec process-input`. Claude reads `input.md` + the full repo and produces:

```markdown
# Processed Analysis: <project name>

## Common Workflows
Step-by-step breakdown of the main user/developer workflows through the codebase.
E.g., "Training a model": config loading → dataset init → model build → training loop → checkpoint.
Each workflow traces the actual code path with function names and files.

## Common Call Graphs
The major call paths, written as trees with qualified function names.
E.g.:
  TrainEvalRunner.fit()
    → MLIPTrainEvalUnit.train_step()
      → HydraModel.forward()
        → backbone.forward()
        → head.forward()
      → loss.compute()

## Test Coverage Analysis
For each test file:
  - What module/functions it covers
  - For each tested function: what properties are verified, what edge cases
  - What is NOT tested (gaps)

Example:
  test_uma.py:
    - Tests HydraModel forward pass with random inputs
    - Tests energy/force head output shapes
    - Gap: empty batch edge case not tested
    - Gap: PBC boundary conditions not tested
```

## The .engspec File Format (Final, Validated)

After the swarm agent fills in and validates the spec:

```markdown
<!-- engspec v1 -->
<!-- source: src/models/backbone.py -->
<!-- language: python -->
<!-- hash: sha256:abc123 -->
<!-- status: validated -->
<!-- validated: 2026-03-14T10:30:00Z -->
<!-- regeneration_count: 5 -->
<!-- regeneration_pass_rate: 5/5 -->

## `forward(self, data: AtomicData) -> dict[str, Tensor]`

### Purpose
Runs the SO(3)-equivariant message passing backbone on atomic graph data,
producing per-node feature embeddings for downstream property heads.

### Context
- Called by: HydraModel.forward()
- Calls: self._build_graph(), self._message_passing_layers()
- Test coverage: tested in test_uma.py (forward pass shape check)

### Preconditions
- data contains: pos [N,3], atomic_numbers [N], edge_index [2,E]
- N >= 1 (at least one atom)
- edge_index values in [0, N)

### Postconditions
- Returns dict with key "node_embedding" of shape [N, hidden_dim]
- Output is equivariant under SO(3) rotations of input positions
- All input tensors are unmodified

### Invariants
- Deterministic for same input (no dropout in eval mode)
- Does not modify self (no state changes during forward)

### Implementation Notes
- Uses block-diagonal Wigner rotation matrices for lmax=2
  (1x1 + 3x3 + 5x5 blocks, not dense 9x9) for efficiency

### Failure Modes
- Raises if edge_index contains values >= N (index out of bounds)
- NaN propagation if input positions contain NaN
- OOM for very large graphs (>100k atoms) on small GPUs

### Test Strategy
- Property: output shape matches [N, hidden_dim] for various N
- Property: equivariance — rotate input, compare with rotated output
- Edge case: single atom (no edges)
- Edge case: self-loops in edge_index

### Debate Log
| Round | Agent | Finding | Ruling | Action |
|-------|-------|---------|--------|--------|

---
```

Specs should on average be **shorter** than the code. Implementation Notes may reference language-specific constructs only when necessary for performance.

## The Agent Swarm

Each agent in the swarm is independent and owns one `.engspec` file. The agent:

1. **Reads**: skeleton `.engspec` + corresponding source file + `input_processed.md`
2. **For each function** in the file:
   a. Fills in all placeholder sections from source code analysis
   b. **Regeneration verification loop**:
      - Re-implement the function from the spec alone (no source), N times
      - Each regeneration is independent (agent sees ONLY the spec)
      - Compare each regeneration against original:
        - Does it preserve all preconditions?
        - Does it satisfy all postconditions?
        - Does it handle all failure modes?
        - Are the core algorithmic properties present?
        - (Does NOT need to be character-identical)
      - If ANY regeneration misses essential properties:
        - Identify what the spec failed to communicate
        - Update the spec with more detail
        - Re-run regenerations
      - Converged when all N regenerations pass
3. **Updates metadata**: status → `validated`, timestamp, pass rate
4. **Returns** the final validated `.engspec` file

```python
# Conceptual — each agent runs this loop per function
async def validate_spec(spec: EngSpec, original_source: str,
                        regen_count: int = 5, max_iterations: int = 5):
    for iteration in range(max_iterations):
        results = []
        for i in range(regen_count):
            regen = await regenerate_from_spec_only(spec)
            result = await compare_regen_vs_original(original_source, regen, spec)
            results.append(result)

        if all(r.passed for r in results):
            return spec  # validated!

        gaps = collect_missing_properties(results)
        spec = await refine_spec(spec, gaps, original_source)

    return spec  # best effort after max iterations
```

## Key Data Models

```python
@dataclass
class EngSpec:
    function_signature: str
    purpose: str
    context: ContextInfo              # callers, callees, test coverage
    preconditions: list[str]
    postconditions: list[str]
    invariants: list[str]
    implementation_notes: list[str]
    failure_modes: list[str]
    test_strategy: list[str]
    debate_log: list[DebateEntry]

@dataclass
class ContextInfo:
    called_by: list[str]
    calls: list[str]
    test_coverage: str                # summary of what's tested

@dataclass
class EngspecFile:
    source_path: str
    language: str                     # python, rust, go, cpp, etc.
    source_hash: str
    status: str                       # "skeleton" or "validated"
    validated: datetime | None
    regeneration_count: int
    regeneration_pass_rate: str       # e.g. "5/5"
    specs: dict[str, EngSpec]         # keyed by function name

    @classmethod
    def parse(cls, path: str) -> EngspecFile: ...
    def serialize(self) -> str: ...

@dataclass
class CallGraph:
    nodes: dict[str, EngSpec]
    edges: list[tuple[str, str]]      # caller → callee (from Context sections)

    def subgraph(self, root: str, depth: int) -> CallGraph: ...
    def topological_order(self) -> list[str]: ...

@dataclass
class SyncReport:
    orphaned_specs: list[str]         # .engspec entries with no matching function
    uncovered_functions: list[str]    # source functions with no .engspec entry
    stale_specs: list[str]            # source hash mismatch
    covered_functions: list[str]

@dataclass
class RegenerationResult:
    attempt: int
    generated_code: str
    language: str                     # language the agent chose
    properties_matched: list[str]
    properties_missing: list[str]
    passed: bool
```

## CLI Commands

```
engspec create-input <repo-path>          # Analyze repo → input.md (human reviews)
  --url <git-url>                         #   clone from URL first

engspec process-input <repo-path>         # input.md + repo → input_processed.md + skeleton .engspec files
  --input <input.md>                      #   source input.md (default: ./input.md)

engspec bootstrap <path>                  # Run agent swarm: fill + verify all .engspec files
  --input <input_processed.md>            #   processed context (required)
  --regen-count N                         #   regenerations per function (default 5)
  --max-iterations N                      #   max spec revision rounds (default 5)
  --parallelism N                         #   max concurrent agents (default 10)

engspec sync <path>                       # Report orphans, uncovered, stale

engspec graph <path> [--format dot|json]  # Build + display call graph from .engspec Context sections

engspec compose <path>                    # Check postcondition→precondition chains

engspec revalidate <path>                 # Re-run regeneration verification on existing .engspec files
```
